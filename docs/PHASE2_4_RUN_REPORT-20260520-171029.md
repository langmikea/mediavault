### 2.4 Cowork-Windows FUSE mount blocks SQLite commit

**Surfaced during execution:** First run of the cleanup script
(operating in-place against `core/mediavault.sqlite`) ran the three
operations successfully but `conn.commit()` raised
`sqlite3.OperationalError: disk I/O error`. Inspection showed
`core/mediavault.sqlite-journal` was left behind (37448 bytes); a
fresh `sqlite3.connect()` afterwards also raised `disk I/O error`
because SQLite's rollback path also failed on the FUSE mount. This is
the same class of issue PHASE2A §3 documented for git's
`.lock`-rename pattern — the Cowork-on-Windows mount does not handle
atomic-rename-and-delete sequences cleanly.

**Resolved without asking** — the recovery path was deterministic:

1. Granted file-delete permission (`mcp__cowork__allow_cowork_file_
   delete`), removed the stale journal.
2. Restored `core/mediavault.sqlite` from the pre-write backup via
   `cp` (SHA-256 of restored file matched the backup exactly — DB
   confirmed pristine pre-write).
3. Re-ran the cleanup against `/tmp/mediavault.work.sqlite` (a copy
   on the VM's native ext4), where SQLite's commit semantics work
   normally.
4. `cp` the post-cleanup `/tmp` DB back over `core/mediavault.sqlite`
   (the same OS-level write PHASE2A §3 used for grown text files).
5. Byte-compared MV-side DB SHA-256 against the `/tmp` SHA-256 —
   identical.

The script was amended pre-cleanup to accept an optional `--db` flag
so the `/tmp` invocation is reproducible. The default path still
points at `core/mediavault.sqlite` for normal (non-Cowork) operator
runs. The amendment is documented in the script's module docstring.

The pre-write backup proved its value here: without it, the
interrupted-commit + failed-rollback state would have been
unrecoverable. **Recommendation for Phase 2.5 and future MV-side
data cleanups:** default to the `/tmp + cp` shape from the start,
the same way PHASE2A §3 defaulted to `cp`-via-bash for git plumbing.

## Appendix — exact SQL used

For posterity, the three statements executed inside the single
transaction, exactly as bound (parameters shown as `?` with their
count). All three operate on the `tags` cache; none writes
`artifacts.tags`, so the §4.5.1(b) rule is unaffected.

```sql
-- Step 1: 69 rows deleted (66 distinct cache-only slugs + 3 duplicates).
-- Parameters: 69 ? placeholders, bound to the 69 distinct slugs from artifacts.tags
-- in sorted order.
DELETE FROM tags WHERE slug NOT IN (?, ?, ?, …);

-- Step 2: 54 INSERTs, one per live-only slug, sorted alphabetically.
-- Each row: usage_count starts at 0 (recomputed in step 3); created_at = now.
INSERT INTO tags(slug, usage_count, created_at) VALUES (?, 0, ?);

-- Step 3: 69 rows updated. Character-identical to handle_tag_merge
-- (imgserver.py:1581) and handle_tag_bulk_delete (imgserver.py:1640).
UPDATE tags SET usage_count = (
  SELECT COUNT(*) FROM artifacts a, json_each(a.tags) j
  WHERE j.value = tags.slug

```
