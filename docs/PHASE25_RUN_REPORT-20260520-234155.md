### 3.2 What the script does

Single SQLite transaction (`BEGIN IMMEDIATE` … `COMMIT`), five steps,
all executed against a `/tmp` copy of the DB (the live file is
overwritten via `cp` only after verification succeeds).

1. **`DROP INDEX IF EXISTS`** for the four obsolete indexes
   (`idx_tags_slug_category`, `idx_tags_slug_when_null_cat`,
   `idx_tags_category`, `idx_tags_proposed`). Must precede the column
   drops — SQLite refuses to `DROP COLUMN` while an index references
   it.
2. **`ALTER TABLE tags DROP COLUMN`** ×4 — `description`, `category`,
   `is_proposed`, `is_exclusive`. Requires SQLite ≥ 3.35.0 (checked at
   the top of `main()`; library is 3.37.2 on this host).
3. **CREATE-INSERT-DROP-RENAME** — SQLite cannot add a `PRIMARY KEY`
   constraint via `ALTER TABLE`. The script creates a fresh
   `tags_new` with the target 4-column shape (`slug TEXT PRIMARY KEY,
   display_name TEXT, usage_count INTEGER NOT NULL DEFAULT 0,
   created_at TEXT`), copies all 69 rows with `INSERT … SELECT`, drops
   the old `tags`, and renames `tags_new → tags`.
4. **Verification inside the open transaction** — `PRAGMA
   table_info(tags)` matches the target 4-column shape; row count is
   69; `SUM(usage_count)` is 453; distinct slugs is 69 (PK enforces
   uniqueness — bookkeeping); the set of slugs is identical to the
   pre-write set (no slug lost or renamed); the only index on `tags`
   is the implicit `sqlite_autoindex_tags_1`; `EXPLAIN QUERY PLAN
   SELECT slug, usage_count FROM tags WHERE slug=?` returns
   `SEARCH tags USING INDEX sqlite_autoindex_tags_1 (slug=?)`.
5. **`COMMIT`** — single transaction throughout. On any exception, the
   `try/except` block calls `conn.rollback()` before re-raising; the
   `/tmp` DB is left in pre-transaction state, the live DB is
   untouched, and the second backup taken in §2.4 of the recovery
   path remains the truth.

The script does **not** write `artifacts.tags`. The §4.5.1(b)
single-writer rule for `artifacts.tags` is unaffected (verified
post-commit at §4.6 below).

## Appendix — exact SQL used

All five statements executed inside one transaction against
`/tmp/mediavault.phase25.sqlite`; the migrated DB was then `cp`ed over
`core/mediavault.sqlite` and SHA-256 verified.

```sql
-- Step 1: drop the four obsolete user indexes.
DROP INDEX IF EXISTS idx_tags_slug_category;
DROP INDEX IF EXISTS idx_tags_slug_when_null_cat;
DROP INDEX IF EXISTS idx_tags_category;
DROP INDEX IF EXISTS idx_tags_proposed;

-- Step 2: drop the four registry-era columns. Requires SQLite >= 3.35.
ALTER TABLE tags DROP COLUMN description;
ALTER TABLE tags DROP COLUMN category;
ALTER TABLE tags DROP COLUMN is_proposed;
ALTER TABLE tags DROP COLUMN is_exclusive;

-- Step 3: add the slug PRIMARY KEY constraint via
-- CREATE-INSERT-DROP-RENAME (SQLite cannot add a PK via ALTER TABLE).
CREATE TABLE tags_new (
  slug         TEXT    PRIMARY KEY,
  display_name TEXT,
  usage_count  INTEGER NOT NULL DEFAULT 0,
  created_at   TEXT
);
INSERT INTO tags_new (slug, display_name, usage_count, created_at)
  SELECT slug, display_name, usage_count, created_at FROM tags;
DROP TABLE tags;
ALTER TABLE tags_new RENAME TO tags;

-- Step 4: verification (read-only) — see §3.2 / §4 for the values.

-- Step 5: COMMIT.
```

End-state shape (69 rows):

- `slug` — PRIMARY KEY. All 69 are namespaced (`namespace:value`),
  per the post-Phase-2.4 invariant; PK enforces uniqueness.
- `display_name` — NULL across all rows. The human label is supplied
  per-namespace by the §5.4 `vocabulary` registry, not per-tag.
- `usage_count` — matches the real count of occurrences in
  `artifacts.tags` JSON arrays for every row; `SUM` = 453.
- `created_at` — preserved from the pre-migration row (15 rows from
  the original cache, 54 from the Phase 2.4 inserts).
